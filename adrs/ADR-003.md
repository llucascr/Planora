# ADR-003: Autenticação via GitHub OAuth2 com JWT RS256 próprio

## Status

Aceito (substitui a decisão anterior de usar sessão com HttpSession)

## Contexto

No início do projeto, tentamos usar sessões gerenciadas pelo servidor com HttpSession após o login OAuth2. Isso funcionou nos primeiros testes, mas criou problemas sérios com o frontend em React, que faz chamadas AJAX e não conseguia manter a sessão corretamente. Além disso, sessões server-side impossibilitam o uso de múltiplas instâncias do backend no futuro.

Durante o desenvolvimento, identificamos também que a duração do token foi definida como um número mágico (860000L segundos, aproximadamente 9,9 dias) no OauthService. Isso precisa ser corrigido para Duration.ofDays(1) como parte da refatoração, tornando a expiração explícita e mais segura.

## Alternativas Consideradas

Sessão com HttpSession no servidor — descartada por ser incompatível com o frontend SPA e com escalabilidade horizontal.

## Decisão

Após o login OAuth2 com o GitHub, o backend gera um JWT próprio assinado com RS256 usando o par de chaves RSA (app.key e app.pub). Esse token carrega o identificador do usuário (claim sub) e o token de acesso ao GitHub (claim githubToken). O frontend armazena o JWT em um cookie de nome token e o envia em cada requisição. O Spring Security valida o token automaticamente via JwtDecoder configurado com a chave pública. A expiração do token será padronizada para 1 dia durante a refatoração.

## Consequências

### Positivas

- Backend completamente sem estado de sessão, o que facilita o escalonamento.
- O token GitHub embutido no JWT evita uma consulta ao banco por requisição.
- A chave pública pode ser compartilhada com outros serviços para validar tokens.
- A padronização da expiração para 1 dia (planejada na refatoração) reduzirá a janela de exposição em caso de vazamento.

### Negativas / Trade-offs

- O token cresce em tamanho por carregar o token do GitHub junto.
- Revogar um token individualmente antes do vencimento exigiria uma lista de bloqueio.
