# ADR-005: Sincronização de Issues via Webhook do GitHub

## Status

Aceito

## Contexto

As issues criadas no Kanban são, na verdade, issues do GitHub. Isso significa que qualquer alteração feita diretamente na interface do GitHub (título, estado, responsáveis) não refletiria no Kanban do Planora sem algum mecanismo de sincronização. A solução mais simples seria fazer polling periódico na API do GitHub, mas isso significaria N chamadas constantes por board ativo, o que é ineficiente e esgota rapidamente os limites de requisição da API.

## Decisão

Ao criar um KanbanBoard vinculado a um repositório, o backend registra automaticamente um webhook no GitHub via GithubClient. O endpoint POST /v1/webhook/github/issues recebe os eventos do GitHub (issue opened, closed, edited, labeled etc.) e o WebhookService atualiza o banco de dados do Planora conforme necessário. Para evitar duplicidade, o sistema verifica se já existe um webhook registrado para o repositório antes de criar um novo, reaproveitando o mesmo entre múltiplos boards do mesmo repositório.

## Consequências

### Positivas

- O Kanban permanece sincronizado com o GitHub sem necessidade de polling.
- Um único webhook por repositório atende múltiplos boards, reduzindo o número de hooks.
- O backend reage a eventos em tempo real, sem custo de requisições periódicas.

### Negativas / Trade-offs

- O token do usuário precisa da permissão admin:repo_hook para registrar webhooks.
- Se o webhook falhar ou for deletado manualmente no GitHub, a sincronização para silenciosamente.
- O endpoint de webhook precisa estar exposto publicamente, o que complica o desenvolvimento local.
